# Example: /accessibility-review

## User Input
Review the signup form accessibility. It has: email field, password field,
confirm password, a "terms of service" checkbox, and a Submit button. There's
also a password strength indicator that changes color (red/yellow/green).

## Skill Output

# Accessibility Review: Signup Form

## Summary
**WCAG Level**: AA target
**Overall**: 3 critical violations found — form is not usable by screen reader users
or keyboard-only users in its current state.

## Findings

### 🔴 WCAG Violation: SC 1.3.1 — Missing form labels
**Level**: A (base level — must fix)
**Location**: Email input, Password input, Confirm Password input
**Problem**: Inputs use placeholder text as the only label. Placeholders disappear
when users start typing — screen reader users lose context mid-form.
**Impact**: VoiceOver announces "text field" with no label. Users cannot tell what
each field is for when tabbing through.
**Current HTML**:
```html
<input type="email" placeholder="Enter your email" />
```
**Fix**:
```html
<!-- Option 1: Visible label (preferred) -->
<label for="email">Email address</label>
<input id="email" type="email" placeholder="name@example.com" />

<!-- Option 2: Visually hidden label (if design requires no visible label) -->
<label for="email" class="sr-only">Email address</label>
<input id="email" type="email" placeholder="name@example.com" />
```

### 🔴 WCAG Violation: SC 1.4.1 — Color as only indicator for password strength
**Level**: A (base level — must fix)
**Location**: Password strength indicator
**Problem**: Strength is shown only by color (red/yellow/green bar). ~8% of users
are color-blind and cannot distinguish these colors.
**Impact**: Color-blind users cannot assess password strength at all.
**Fix**: Add text label alongside color indicator:
```html
<div class="strength-indicator">
  <div class="strength-bar" data-level="weak"></div>
  <span class="strength-label">Weak — add numbers and symbols</span>
  <!-- Screen reader also reads this text -->
</div>
```

### 🔴 WCAG Violation: SC 1.3.1 — Error messages not associated with inputs
**Level**: A
**Location**: Validation errors shown below inputs
**Problem**: Error messages are visually below their fields but not programmatically
associated. Screen readers announce errors in document order, then the user must
figure out which error belongs to which field.
**Current**:
```html
<input type="email" />
<span class="error">Please enter a valid email</span>
```
**Fix**:
```html
<input type="email" id="email"
       aria-describedby="email-error"
       aria-invalid="true" />
<span id="email-error" role="alert" class="error">
  Please enter a valid email address
</span>
```

### 🟡 Improvement: Terms checkbox needs descriptive label
**Location**: Terms of Service checkbox
**Current**:
```html
<input type="checkbox" id="tos" />
<label for="tos">I agree to the <a href="/terms">terms</a></label>
```
**Issue**: Screen readers read "I agree to the terms" but the link context is lost.
**Better**:
```html
<label for="tos">
  I agree to the
  <a href="/terms" target="_blank">Terms of Service (opens in new tab)</a>
</label>
```

### 🟡 Improvement: Submit button disabled state unexplained
**Location**: Submit button (disabled until TOS checked)
**Problem**: Button is disabled before TOS checked, but nothing tells users WHY.
Screen readers just hear "Submit, dimmed" with no explanation.
**Fix**: Add `aria-describedby` pointing to a hint:
```html
<span id="submit-hint" class="sr-only">
  Check the terms of service checkbox to enable this button
</span>
<button type="submit" disabled aria-describedby="submit-hint">
  Create Account
</button>
```

### 💭 Best Practice: Add autocomplete attributes
Helps password managers and mobile autofill:
```html
<input type="email" autocomplete="email" />
<input type="password" autocomplete="new-password" />
<input type="password" autocomplete="new-password" />  <!-- confirm -->
```

## Quick Wins (< 30 minutes)
1. Add `<label>` elements to all 3 inputs — High impact, 10 min
2. Add `aria-describedby` to link errors to inputs — High impact, 15 min
3. Add text to password strength indicator — High impact, 10 min
4. Add `autocomplete` attributes — Medium impact, 5 min

## Testing Checklist
- [ ] Tab through form with keyboard only — can you reach and activate everything?
- [ ] Enable VoiceOver (Mac: Cmd+F5) — does it announce labels when focusing fields?
- [ ] Submit with empty fields — does VoiceOver announce the errors?
- [ ] Run `axe` browser extension — should show 0 violations after fixes
- [ ] Test with Chrome's color-vision deficiency emulator (DevTools → Rendering → Emulate vision deficiencies)

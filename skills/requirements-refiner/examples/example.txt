# Example: /requirements-refiner

## User Input
"Add social login to the app. Users should be able to sign in with Google and maybe
Facebook. It should be easy and secure. We want to increase signups."

## Skill Output

# Requirements Specification: Social Login

## Problem Statement
New user registration has a 68% drop-off at the signup form (email + password).
Users expect one-click sign-in via existing social accounts. Goal: reduce signup
friction and increase completed registrations.

## User Stories
- As a new user, I want to sign up with my Google account so that I don't have
  to create and remember a new password
- As a returning user, I want to sign in with my existing social account so that
  I can access my account without remembering my credentials
- As a user who previously signed up with email, I want to link my Google account
  so that I can use either method to sign in

## Functional Requirements

### REQ-1: Google Sign-In
**Priority**: Must-have
**Description**: Users can authenticate using their Google account via OAuth 2.0
**Acceptance Criteria**:
- GIVEN a new visitor WHEN they click "Continue with Google" THEN they see
  Google's OAuth consent screen
- GIVEN a user completes Google OAuth WHEN they have no existing account THEN
  a new account is created using their Google profile (name, email, avatar)
- GIVEN a user completes Google OAuth WHEN they have an existing email account
  THEN their accounts are linked and they are signed in
- GIVEN a user previously signed in with Google WHEN they click "Continue with Google"
  THEN they are signed in without seeing the consent screen (already authorized)
**Edge case**: What if Google email matches an existing account created with a
different social provider? → [OPEN QUESTION #1]

### REQ-2: Facebook Sign-In (conditional)
**Priority**: Should-have (only if analytics show demand)
**Description**: Same OAuth flow as Google, via Facebook Login
**Note**: Requires Facebook App Review for `email` permission — 2-4 week lead time.
Recommend starting with Google only, adding Facebook in a follow-up iteration.

### REQ-3: Account Linking
**Priority**: Must-have (required for REQ-1 to work safely)
**Description**: Users with existing email accounts can link social providers
**Acceptance Criteria**:
- GIVEN a logged-in user WHEN they visit Account Settings THEN they see
  "Connected accounts" with option to link/unlink Google
- GIVEN a user links Google WHEN they already have Google linked THEN show error
  "Google account already connected to another user"

## Non-Functional Requirements
- **Security**: Must use OAuth 2.0 PKCE flow (not implicit). State parameter required to prevent CSRF.
- **Privacy**: Only request `openid`, `email`, `profile` scopes — minimum necessary
- **Performance**: OAuth redirect roundtrip must complete < 3 seconds (user-perceived)
- **Availability**: Social login failure must fall back gracefully to email/password — not block login entirely

## Constraints & Assumptions
| Type | Statement | Risk If Wrong |
|------|-----------|---------------|
| Assumption | Users' Google email = their primary identity | Users with multiple Google accounts may get confused |
| Assumption | Facebook is lower priority than Google | Validate with user research before deprioritizing |
| Constraint | Must work on iOS Safari (no third-party cookies) | Standard OAuth redirect flow required, not popup |
| Constraint | GDPR compliance required | Need to handle data deletion requests for social-linked accounts |

## Out of Scope
- Apple Sign-In (required for iOS App Store — add to next sprint if building native app)
- SSO/SAML for enterprise customers (separate product requirement)
- Social sharing or posting on behalf of users (different OAuth scopes entirely)
- Username display from social profile (use email as display name for now)

## Open Questions
| # | Question | Impact | Needed By |
|---|----------|--------|-----------|
| 1 | Account collision: Google email matches existing email account — link automatically or ask user? | Blocks REQ-3 design | Before dev starts |
| 2 | Do we store Google profile photo? Where? | Affects storage design | Before dev starts |
| 3 | What happens when user revokes Google app permissions? | Affects session management | Before QA |

## Risks
| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Google changes OAuth API | Low | High | Abstract behind auth adapter layer |
| Social login becomes unavailable (Google outage) | Low | High | Email/password always available as fallback |
| Facebook App Review delays | High | Medium | Launch Google only; Facebook in v2 |

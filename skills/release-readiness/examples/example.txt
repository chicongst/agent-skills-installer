# Example: /release-readiness

## User Input
We're releasing v2.0 of our e-commerce platform next Thursday. Changes include:
a new checkout flow, price recalculation engine rewrite, 3 DB migrations (add columns,
add index, backfill), and dropping support for API v1. Expected traffic: Black Friday
levels (~3x normal). Team: 5 engineers, 2 QA.

## Skill Output

# Release Readiness: v2.0 E-Commerce Platform

## Scope
- **Changes**: New checkout flow, price engine rewrite, 3 DB migrations, API v1 sunset
- **Risk level**: 🔴 HIGH — revenue-critical path, breaking changes, data migrations
- **Blast radius**: All checkout flows, all integrations using API v1, full user base

## Pre-Release Checklist

### Code & Testing
- [x] New checkout flow QA-approved on staging
- [ ] Price engine: regression tested against 10k historical orders — **PENDING** (ETA Wed)
- [ ] Load test at 3x normal traffic — **NOT STARTED** (must complete before Thursday)
- [ ] API v1 sunset: confirm all known integrations migrated to v2

### Database Migrations
- [ ] Migration 1 (add columns): Tested on staging DB with production-size dataset — confirm no table locks
- [ ] Migration 2 (add index): Tested with CONCURRENTLY flag — confirm duration < 5 min
- [ ] Migration 3 (backfill): Tested in batches — confirm throughput and estimated runtime
- [ ] All migrations: rollback procedures documented and tested

### Observability
- [ ] Dashboard updated with new checkout funnel metrics
- [ ] Alerts configured: checkout_error_rate > 1%, payment_failure_rate > 0.5%
- [ ] Price engine: latency alert if p99 > 200ms

### Communication
- [ ] API v1 deprecation notice sent to all integration partners (at least 2 weeks ago — check)
- [ ] On-call team briefed and available Thursday + Friday
- [ ] Customer support briefed on new checkout UX changes
- [ ] Rollback runbook distributed to all engineers

## 🚫 BLOCKERS (Must resolve before release)

1. **Price engine regression tests not complete** — Cannot release a rewritten pricing
   engine without full regression validation. ETA Wednesday; if it slips, release must slip.

2. **Load test not started** — 3x traffic on Black Friday with a new checkout flow is
   extremely high risk. Load test must complete and pass before Thursday.

3. **API v1 partner migration unconfirmed** — Dropping v1 without confirming all partners
   have migrated will break live integrations. Get written confirmation from each partner.

## Rollout Plan

```
Thursday 10:00 UTC — DB migrations (run before code deploy)
  ├── Migration 1: Add columns (~2 min, no lock expected)
  ├── Migration 2: Add index CONCURRENTLY (~8 min, monitor)
  └── Migration 3: Backfill in batches (monitor for ~30 min)

Thursday 10:45 UTC — Code deploy
  ├── Deploy to 10% canary
  ├── Monitor: checkout_error_rate, payment_success_rate, price_latency (15 min)
  ├── If healthy → 50% (monitor 15 min)
  └── If healthy → 100%

Thursday 11:30 UTC — API v1 sunset
  └── Only after 100% rollout confirmed healthy
```

## Rollback Plan

**Trigger**: checkout_error_rate > 2% OR payment_failure_rate > 1% sustained 5 minutes

**Code rollback** (5 min):
```bash
kubectl rollout undo deployment/ecommerce-api
# Validate: curl /health → old version header
```

**Migration rollback**:
- Migration 1 (add columns): `ALTER TABLE orders DROP COLUMN IF EXISTS ...` — safe
- Migration 2 (add index): `DROP INDEX CONCURRENTLY ...` — safe, ~2 min
- Migration 3 (backfill): No rollback needed — backfill is additive

**API v1 rollback**: Re-enable v1 routes in feature flag — instant.

**Data impact**: New orders created during v2 window are compatible with v1 code.
No data loss on rollback.
